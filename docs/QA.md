# 学习问答整理（QA）

这份文档记录了构建 microLLM 过程中提出的所有疑问与解答，按主题归类，方便回顾。
**核心心法**：tokenizer 管"切成什么单位"，模型管"如何预测下一个"，attention 管"如何结合上下文"——三件事互相独立。

---

## 一、基础概念

### Q：Transformer 架构是什么？
一种神经网络架构，核心是 **Attention（注意力）** 机制。它的任务是"给定前文，预测下一个 token"。GPT = Generative（生成式）+ Pretrained（预训练）+ Transformer。
数据流：文字 → 分词(token) → Embedding+位置编码 → 多层(Attention + 前馈网络) → 输出层 → 下一个词的概率。

### Q：张量（tensor）是什么？
**带类型和形状的多维数组**，可理解为"会做数学、能上 GPU、能自动求导的 NumPy 数组"。
- 0维=标量，1维=向量，2维=矩阵，n维=张量（统称）。
- `shape` 是每个维度的大小，如 `(4, 8)`=4行8列。
- 相比 Python list 的优势：① 向量化运算快；② 能上 GPU 并行；③ **自动求导（autograd）**——会记录经历的每步运算，反向传播时自动算梯度。
- `dtype`：`torch.long`(整数，存token编号)、`torch.float`(小数，存权重)。

### Q：神经网络是什么？
**一个"参数可调的函数"**，我们用数据把它的参数（旋钮）调到能完成任务。
- 最小单位**神经元**：把输入各乘一个**权重**、加起来、再加**偏置** → 一个加权求和。
- **学习 = 调这些权重和偏置**。

### Q：为什么是"加权求和"？
加权求和就是**可定制的打分公式**——每个因素配一个权重表示"它有多重要"。神经网络自动找出这些权重，而不是人去拍脑袋设。单个很简单，但成千上万个叠起来 + 非线性，就能逼近几乎任何复杂函数。

### Q：线性层、归一化是什么？
- **线性层（nn.Linear）**：一排神经元，本质是"矩阵乘法 + 加偏置"`输出 = 输入×W + b`，是最基础的积木。
- **归一化（LayerNorm）**：把中途的数值重新缩放回健康范围，让训练稳定。Transformer 阶段才用到。

## 二、神经网络如何学习

### Q：学习是"随机调权重看达没达到预期"吗？
框架对，但**不是随机调，是有方向地调**：
1. 前向算出预测 → 2. 算 **loss**（差多少）→ 3. **反向传播**算出每个权重的下坡方向 → 4. 朝下坡各挪一小步 → 重复几百万次。
比喻：蒙眼下山，loss 是海拔，梯度告诉你哪边下坡，一步步走到谷底。

### Q：这么多参数，怎么知道要调哪个？
**全都调，不用挑**。数学（梯度）会一次性算出每个旋钮的方向。
- 一个权重对 loss 的影响力 = 它前面乘的那个输入值。
- **反向传播（backprop）**：从 loss 倒推、用链式法则一遍算出所有参数的梯度，计算量只相当于前向的两三倍。
- PyTorch 自动做：`loss.backward()` 填好每个参数的 `.grad`，`optimizer.step()` 执行更新。

### Q：优化器（optimizer）到底做了什么？
**专门负责"根据梯度更新权重"**。
- `loss.backward()` 只**算方向**（填 `.grad`），不动权重。
- `optimizer.step()` 才**真正动权重**，最简单公式（SGD）：`新权重 = 旧权重 − 学习率 × 梯度`（减是因为梯度指向上坡，要下坡）。
- 我们用 **AdamW**：SGD 的升级版，加了"动量"和"每参数自适应步长"，但核心仍是"按梯度更新"。

### 三件套顺序（训练循环核心）
```python
optimizer.zero_grad()  # 清空上一轮旧梯度（默认会累加）
loss.backward()        # 算方向：填 .grad，不动权重
optimizer.step()       # 迈步：用 新=旧−lr×grad 更新所有参数
```

## 三、Tokenizer 与数据

### Q：token 不是单词或单词的一部分吗？为什么我们拆成单个字母？
token 的**粒度是设计选择**：字符级 / 整词级 / 子词级。
- 真实大模型用**子词级（BPE）**：常见词自成一个token，罕见词拆成几块，兼顾词表大小和无OOV。
- 我们用**字符级**纯粹为教学：词表只65、五行写完、省去BPE复杂度，让你专注Transformer。
- 原理完全一样，**粒度可插拔，架构才是核心**。

### Q：tokenizer 要用类包裹吗？
**要**。因为它有状态（stoi/itos/vocab_size），且**训练和生成必须用同一个词表**。类把"状态+方法"打包，并保证一致性。
- 用法：`tok = Tokenizer(text)`（实例化，自动调 `__init__`）→ `tok.encode("hi")`。
- ⚠️ 不要手动调 `Tokenizer.__init__()`，那是 `Tokenizer(text)` 隐式触发的。

### Q：get_batch 的作用是什么？
从一整条长文本流里**随机剪下几段，加工成 (输入x, 答案y)**。
- `x` = `data[i : i+block_size]`，`y` = `data[i+1 : i+block_size+1]`（**y 是 x 右移一位**）。
- 一段长 block_size 的序列藏着 block_size 个训练例子（每个位置都是一个"看前文猜下一字"）。
- 随机选起点 → 防止记住固定顺序；批量(batch) → 训练更快更稳。
- 起点范围要限制在 `len(data) - block_size`，否则末尾越界。

### Q：训练集/验证集用什么存储？Python 不支持多返回值？
**支持**——靠**元组**。`return a, b` 把两者打包成一个元组，`x, y = func()` 再解包。
train_data / val_data 就是两个 `torch.Tensor`（张量切片返回的还是张量，且是不复制的视图）。

## 四、模型与预测

### Q：model.py 是干嘛的？
整个项目分三块：tokenizer/data(传送带) → **model(会学习的大脑)** → train(教练)。
model 定义大脑长什么样：① `__init__` 准备参数（那张表）；② `forward` 写"输入→预测"的计算规则。必须继承 `nn.Module`。

### Q：nn 模块是做什么的？
`torch.nn` = 搭神经网络的**零件箱**：① 各种层（Embedding/Linear/LayerNorm，自带可学习参数）；② 基类 `nn.Module`（所有模型继承它）；③ 损失函数。
`F`（`torch.nn.functional`）= 同一批操作的**即用即弃函数版**（如 `F.softmax`、`F.cross_entropy`），适合无参数的纯计算。

### Q：预测到底是怎么做的？
**不是直接吐一个字，而是给词表里65个字符每个打分**（分数越高越可能）。
- Bigram：内部一张 65×65 表，输入字符→取对应行→得到65个分数(logits)→softmax变概率。
- 表里初始是随机数→预测是乱码；训练不断微调表格数字→变成真实语言规律。
- GPT 和 Bigram 区别只在"怎么得到那排分数"（GPT 用 attention 揉合全部上文），"打分→softmax→概率"框架完全相同。

### Q：`logits = self.token_embedding_table(idx)` 做了什么？logits 是什么类型？
**查表取行**：`token_embedding_table` 是65×65的float表，`idx` 里每个整数编号去取对应那一行（65个float分数）。`idx` 形状 `(B,T)` → 输出 `(B,T,65)`。
- **logits 是 float**（float32）；token编号是 int(long)。身份用整数，数值用小数。

### Q：B、T、C 是什么？为什么 reshape？
logits 形状 `(B,T,C)`：**B**=Batch(几条序列)、**T**=Time(每条几个位置)、**C**=Channels(每位置几个分数=vocab_size)。
- `cross_entropy` 只认二维 `(预测点数, 类别数)`，所以把 `(4,8,65)` 用 `view` 压成 `(32,65)`，targets `(4,8)` 压成 `(32,)`。
- 即把 "4条×8位置" 摊平成 "32个独立预测点"。

### Q：取 B 条序列每条 T 个 token，得到 B 组预测吗？
**得到 B×T 组**！每条序列的每个位置都给一组预测。
- 训练时 B×T 组全用（数据利用率高）；生成时只取每条**最后一个位置**的预测（`logits[:, -1, :]`）。

### Q：targets 是什么？谁给的？
就是**真实答案 yb**（每个位置的下一个字符）。
- 训练时 `model(xb, yb)` 由 train.py 传入 → 算 loss。
- 生成时 `model(idx)` 不传 → `targets=None` → 不算 loss。
- `= None` 是默认值，让同一个 forward 兼顾两种场景。

### Q：`model(xb, yb)` 这方法我没写过？
写过——它就是 `forward` 的简写。`nn.Module` 让"把对象当函数调用"自动转发到 `forward`。`model(xb,yb)` ≡ `forward(idx=xb, targets=yb)`。约定一律用 `model(...)`，别直接调 `.forward()`。

## 五、生成（generate）

### Q：generate 怎么写？也要用类包裹吗？
**逻辑写成 Model 的方法**（它依赖 self、要调 self(idx)），不需要新类。
`generate.py` 文件是另一回事——它是"加载模型→生成→打印"的**入口脚本**（类比 controller）。逻辑在方法里，脚本只是调用它。

### Q：为什么把分数变成概率？什么是分数？
- **分数(logits)** = 模型吐的原始数值，范围任意、可负、加起来≠1，只表达相对大小。
- **softmax** 把它变成合格概率：①指数化(全变正、拉大差距) ②归一化(和为1)。
- 为什么要概率？因为下一步要**按概率抽签**，抽签需要"每个选项占多大比例"。

### Q：为什么按概率抽取，而不直接选最高分？
直接选最高分(argmax)会让生成**死板、重复、卡循环**（同输入永远同输出）。
**按概率采样**：高概率字更常被选，低概率偶尔出现 → 文本更自然有变化。（进阶的 `temperature` 就是调这个随机性大小。）

## 六、超参数与训练

### Q：batch_size、block_size 对计算有什么影响？
两者影响**完全不同**：
| 参数 | 影响 | 类别 |
|---|---|---|
| **block_size** | 模型能看多远的上下文（真实大模型的"上下文窗口"就是它）；对 Bigram 不影响能力，只影响每段切几个训练点 | **模型能力** |
| **batch_size** | 每步并行几条→速度；梯度是多少例子的平均→稳定性(越小越抖)；越大越吃内存 | **训练过程** |

口诀：block_size 管能力，batch_size 管训练稳不稳。

### Q：为什么 loss 在 2.4~2.6 之间跳，不平滑下降？
每步 loss 是从一个**随机 batch** 算的，不同 batch 难度不同所以抖动。看整体趋势(4.6→2.4)而非单步。想看平滑曲线可定期在验证集上算平均 loss(estimate_loss)。

### Q：为什么停在 2.4 不再降？
**Bigram 的天花板**——它只看前1个字符，信息太少。
对照：随机瞎猜 ln(65)≈4.17 → Bigram≈2.4 → Transformer≈1.5（loss 越低预测越准）。

### Q：那个 4.9TB 内存报错是怎么回事？
不是真缺内存，是某维度填错把张量造成天文数字。`4976415100944 ÷ 4 ≈ 1115394²`，而 1115394 是文本长度——说明 `nn.Embedding` 尺寸被误填成 `len(data)` 而非 `vocab_size(65)`。修复：`Model(tok.vocab_size)`。
**经验**：离谱大小的报错，先把数字反推，往往直接指出哪个维度填错。

## 七、Attention（自注意力）

### Q：Attention 是更合理的分词方法吗？
**不是！分词和 attention 完全无关、互相正交。**
- 分词：把文本切成单位（模型之前的预处理）。
- Attention：模型内部如何**结合上下文**。
升级到 Transformer，分词器一字不改（仍字符级）。
- **Bigram**：预测时只看前1个字符（信息瓶颈）。
- **Attention**：回看前面**所有**字符，并按相关性**加权**利用。
一句话：让每个位置"环顾"它前面所有字、按相关性吸收信息，再预测下一字。

### Q：怎么"看上文"？wei 是什么？
**"看上文" = 让每个位置按权重 wei 把前面所有位置的向量加权混合进来。** wei = weights(权重)。
- `wei[t]` = 位置 t 该从前面每个位置吸收多少信息。
- Bigram 相当于 `wei[3]=[0,0,0,1]`(只看自己)；平均=`[0.25,0.25,0.25,0.25]`；Attention=`[0.1,0.6,0.2,0.1]`(模型自己算的、有侧重)。

### Q：`wei @ x` 是什么运算？x 是什么？
`@` = **矩阵乘法**。
- `x`：所有位置的向量堆成的矩阵，形状 `(T, C)`，每行=一个位置的向量(来自 embedding)。
- `wei @ x` 的**第 t 行 = 用 wei 第 t 行的权重，对 x 的所有行加权求和** = 该位置混合了上文的新向量。
- `wei` 是**下三角**矩阵(非零在左下角)：右上=未来=0，保证不看未来(因果掩码)。

### Q：嵌入维度 C 决定什么？是分数种类吗？
**不是**。要分清两个维度：
- **vocab_size=65**：字符种类数，在**输出端**，决定 logits 有几个分数。
- **C / n_embd（如32）**：每个 token 内部用几个数字表示，在**内部**，决定表示的丰富度（"思考草稿纸"的宽度）。C 越大容量越大、计算越贵，但**不改变输出分数个数**。

### Q：Q/K/V 怎么算出来的？三个矩阵哪来的？
每个位置的向量 x，分别乘**三个不同的学习矩阵**得到三个"视角"：
```
Q = x @ Wq   "我想找什么"
K = x @ Wk   "我能提供什么"
V = x @ Wv   "我实际给的内容"
```
- 计算就是矩阵乘法(线性变换)，代码里 Wq/Wk/Wv 就是三个 `nn.Linear(n_embd, head_size, bias=False)`，`self.query(x)` ≡ `x @ Wq`。
- **Wq/Wk/Wv 是可学习参数**：初始随机，靠**梯度下降**训练出来。

### Q：亲和度 `Q·K` 是什么？
**向量点积** = 相似度。`亲和度(t,j) = Q_t · K_j`。
- 点积大且正 → t想找的和j能给的很匹配 → 该多关注。
- 实际是先把 x 变换成 Q、K 再点积，衡量"t想要的 vs j提供的"。

### Q：mask 未来 + softmax 到底在干嘛？
把第1步算出的**原始亲和度分数**，变成**合法权重**(不看未来、每行和为1)。
- 原始分数问题：①含未来位置 ②加起来≠1，没法当混合比例。
- **mask**：未来位置(上三角)设 `-inf`。因为 `e^(-inf)=0`，softmax 后权重恰好为0，干净排除。
- **softmax**：指数化+归一化 → 每行和为1的概率。
- 例：`[3.0, 1.0, 9.0(未来)]` → mask → `[3.0, 1.0, -inf]` → softmax → `[0.88, 0.12, 0.00]`。

### Q：Attention 完整流程（以 "I love eat appl" 15 token 为例）
```
分词+embedding → x (1,15,8)         15个位置，每个8维向量
  ├@Wq→Q ├@Wk→K └@Wv→V (各 1,15,8)
Q @ K^T            → 亲和度 (1,15,15)   ← 是的，会形成15×15矩阵！谁关注谁
×缩放 + mask未来 + softmax → wei (1,15,15) 下三角、每行和1
wei @ V            → out (1,15,8)        每个位置带上语境
→lm_head→ logits (1,15,65) → 取最后一行(位置14) → softmax → 采样 → 'e'
```
关键：①15×15矩阵=`Q@K^T`；②它给 `@V` 做加权混合；③训练时15个位置全用，生成时只取最后一行。

### Q：为什么除以 √head_size（缩放）？
点积随 head_size 增大而变大，数值过大会让 softmax "非黑即白"(一个权重≈1其余≈0)，训练不稳。除以 `√head_size` 拉回合理范围。这就是论文 **"Scaled Dot-Product Attention"** 的 "Scaled"。

### Q：k.transpose(-2,-1) 为什么需要？
要算 `Q @ K^T`。Q、K 都是 `(B,T,hs)`，直接乘形状对不上。把 K 最后两维转置成 `(B,hs,T)`，则 `(B,T,hs)@(B,hs,T)=(B,T,T)`。`-2,-1`=倒数两维，不动 batch 维。

### Q：register_buffer 是什么？
登记一个"**不是参数(不学习)、但属于模型**"的常量张量(如因果掩码 tril)。好处：会随 `model.to(device)` 一起搬，但不被优化器更新。

### Q：多头注意力里的 `n_embd` 是什么？
`n_embd` 是每个 token 在模型内部的**向量宽度**，也就是"思考空间"有多少个数字。
- 它不是词表大小。`vocab_size=65` 表示最终要给65个字符打分；`n_embd=32` 表示每个字符先被翻译成32维向量，在模型内部加工。
- 进入模型后常见形状是 `(B, T, n_embd)`：B=几条序列，T=每条几个 token，n_embd=每个 token 的内部表示宽度。
- 多头注意力通常把 `n_embd` 平分给多个头。例如 `n_embd=32, num_heads=4`，则每个头 `head_size=8`，4个头拼回去仍是32维。

### Q：`torch.cat` 是什么方法？多头里为什么用它？
`torch.cat` 是 PyTorch 的**拼接张量**方法：把多个形状兼容的张量沿某个维度接在一起。
多头注意力里每个头输出一块特征：
```python
Head1(x) -> (B, T, 8)
Head2(x) -> (B, T, 8)
Head3(x) -> (B, T, 8)
Head4(x) -> (B, T, 8)
```
用 `torch.cat([...], dim=-1)` 沿最后一维拼接：
```python
out = torch.cat([head(x) for head in self.heads], dim=-1)
```
结果就是 `(B, T, 32)`。`dim=-1` 表示最后一维，也就是每个 token 的向量维度。

### Q：`self.proj = nn.Linear(n_embd, n_embd)` 是什么属性？
`self.proj` 是多头注意力最后的**输出投影层**。它本质还是一个可学习的线性层，形状是 `n_embd -> n_embd`。
- 多个头 `cat` 后只是把不同头的结果并排放在一起，`proj` 负责再混合一次，让不同头的信息能互相交流。
- 输入输出维度都写 `n_embd`，是为了让注意力模块整体保持形状不变：输入 `(B,T,n_embd)`，输出还是 `(B,T,n_embd)`。
- 它里面有可学习参数，会被 `model.parameters()` 收集，训练时跟 Q/K/V 一起通过梯度下降更新。
- 教学版里暂时可以不用 `proj` 也能跑，但标准 Transformer/GPT 都会保留它，因为它让"拼起来的多头结果"变成一个重新融合后的表示。

## 八、把 Attention 接进模型（迷你 Transformer）

### Q：模型结构为什么要变？n_embd 工作空间是什么？
中间加了 attention，需要一个**比"65个分数"更丰富的思考空间**(n_embd维)来加工含义。
```
idx → token_emb(65→32) ─┐
                        (+) → x(B,T,32) → Head(32→32) → lm_head(32→65) → logits
位置 → pos_emb(→32) ────┘
```
- **入口 65→32**：把"哪个字"翻译成思考向量(token_embedding)。
- **中间 32→32**：在思考空间加工(attention)。
- **出口 32→65**：思考结果翻译回"选哪个字"(lm_head)。
- 类比：解题先在草稿纸推导(n_embd)，最后誊写答案(65)。Bigram 是"看一眼直接猜答案"，Transformer 是"先想清楚再答"。

### Q：为什么需要位置编码(position embedding)？
**因为 attention 不知道顺序**！它靠内容算亲和度，把上文当成一个"集合"，"狗咬人"和"人咬狗"在纯attention看来一样。
- 解决：给每个位置也配一个可学习向量，加到 token 向量上：`x = token_emb + pos_emb`。
- 这样每个位置 = "内容含义 + 在第几位"，模型才知道语序。

### Q：`x = self.sa_head(x)` 和 `logits = self.lm_head(x)` 干嘛？
- `sa_head(x)`：跑注意力头(你写的Head)，每个位置混入上文，32→32，**内容变了但维度不变**。
- `lm_head(x)`：`nn.Linear(n_embd, vocab_size)`，把思考向量映射成65个字符分数，32→65。
- 对照 Bigram：以前一步 `embedding(idx)` 直接出分数；现在拆成"先思考(sa_head)、再出分数(lm_head)"。

### Q：generate 里为什么加 `idx[:, -block_size:]`？
位置编码只有 block_size 个位置。生成时 idx 越来越长，一旦超过 block_size，`arange(T)` 越界报错。所以每次只喂**最后 block_size 个 token**。Bigram 无位置编码故不用裁剪。

### 结果
单头单层 attention 后 loss 从 Bigram 的 ~2.4 降到 ~2.3，生成文本开始出现剧本结构、换行、真单词碎片。提升有限是因为单头/单层/block_size仅8——后续加多头、FFN、残差+LayerNorm、堆叠多层可降到 ~1.5。

## 九、FeedForward（前馈网络）

### Q：FFN 是每个 token 自行单独处理吗？
是的。FeedForward / FFN 不负责 token 之间互相看，它对每个位置的向量**独立使用同一套小神经网络**。
如果输入是 `(B, T, n_embd)`，FFN 处理的是最后一维：
```python
第1个位置的 n_embd 维向量 -> FFN -> 新的 n_embd 维向量
第2个位置的 n_embd 维向量 -> FFN -> 新的 n_embd 维向量
第3个位置的 n_embd 维向量 -> FFN -> 新的 n_embd 维向量
```
注意：这里的"单独处理"不是说只看一个原始字符本身，而是处理这个位置**已经被 embedding、位置编码、attention 加工过的上下文向量**。Attention 已经把上文信息混进来了，所以每个位置的向量里已经带着上下文。

### Q：可是为什么还要处理？处理什么？
Attention 的主要工作是**交流信息**：让每个位置从前面的 token 里按权重吸收信息。FFN 的主要工作是**加工信息**：把这个位置已经拿到的上下文特征重新组合、筛选、放大或压回去。

可以这样分工：
```text
Attention: 我应该从哪些上文位置拿信息？
FFN: 我拿到这些信息之后，应该怎样加工成更有用的内部表示？
```

举个抽象例子，一个位置经过 attention 后的 32 维向量里，可能已经混着这些信息：
```text
当前字符是什么
它在第几个位置
前面是否像一个人名
前面是否刚出现换行
附近是否像一句台词
```
FFN 做的就是在这 32 个数字之间再做一轮可学习的组合。比如它可以学到："如果前面像人名 + 刚换行，那么这个位置更像台词开头"，然后把相关特征变强。

### Q：只有一个 token 参与，怎么完成加工？
因为它处理的不是一个孤零零的 token 编号，而是一个 `n_embd` 维向量。一个向量里有很多通道，每个通道都是模型内部学出来的特征。FFN 可以在这些通道之间做矩阵乘法和非线性变换：
```python
nn.Linear(n_embd, 4 * n_embd)
nn.ReLU()
nn.Linear(4 * n_embd, n_embd)
```
以 `n_embd=32` 为例：
```text
32维 -> 128维 -> ReLU -> 32维
```
- 第一层 `Linear`：把32个特征混合并扩展到128个特征，给模型更大的加工空间。
- `ReLU`：加入非线性，让模型不只是做简单加权平均，而能学到"满足某些条件才激活"的模式。
- 第二层 `Linear`：把128维再压回32维，保持模块输入输出形状一致。

### Q：FFN 公式里的 `W1/W2/b1/b2` 是什么？谁给的？
FFN 常写成：
```text
FFN(x) = ReLU(x @ W1 + b1) @ W2 + b2
```
这里的 `W1/W2` 是两层线性层的**权重矩阵**，`b1/b2` 是两层线性层的**偏置向量**。它们都是模型参数，会被训练更新。

对应到 PyTorch 代码：
```python
self.net = nn.Sequential(
    nn.Linear(n_embd, 4 * n_embd),  # 这一层内部有 W1 和 b1
    nn.ReLU(),
    nn.Linear(4 * n_embd, n_embd),  # 这一层内部有 W2 和 b2
)
```

以 `n_embd=32` 为例，按数学写法理解：
```text
x:  (B, T, 32)
W1: (32, 128)
b1: (128)
ReLU 后: (B, T, 128)
W2: (128, 32)
b2: (32)
最终输出: (B, T, 32)
```

谁规定它们？
- **形状由架构设计规定**：你写 `nn.Linear(n_embd, 4 * n_embd)`，就规定了第一层是 `32 -> 128`；写 `nn.Linear(4 * n_embd, n_embd)`，就规定了第二层是 `128 -> 32`。
- **初始数值由 PyTorch 随机初始化**：创建 `nn.Linear(...)` 时，PyTorch 会自动生成一组小随机数作为 `W`，再生成 `b`。
- **最终数值由训练学出来**：`loss.backward()` 算梯度，`optimizer.step()` 更新这些 `W/b`，所以它们会慢慢变成适合预测下一个 token 的参数。

所以 `n_embd` 确实参与决定了 `W1/W2` 的尺寸，但它不决定矩阵里的具体数字。具体数字不是人工填写的，是模型训练出来的。

### Q：ReLU 为什么叫非线性激活？
`ReLU` 的公式是：
```text
ReLU(z) = max(0, z)
```
小于0的值变成0，大于0的值保留。它不是简单的矩阵乘法，所以叫**非线性**。没有 ReLU 的话，两层 Linear 连起来仍然等价于一层 Linear，表达能力很有限；加了 ReLU 后，FFN 才能学到"满足某些条件就激活，否则关掉"这类更复杂的模式。

一句话记忆：**Attention 负责 token 之间交换信息；FFN 负责每个 token 内部整理信息。**

## 十、调试经验

### Q：4.9TB 内存报错怎么回事？
不是真缺内存，是某维度填错把张量造成天文数字。`4976415100944÷4≈1115394²`(=文本长度²)，说明 `nn.Embedding` 尺寸被误填成 `len(data)` 而非 `vocab_size`。**经验：离谱大小的报错先把数字反推，常能直接定位填错的维度。**

### Q：`Model(n_embd=32, block_size=8, tok.vocab_size)` 报错？
**位置参数不能跟在关键字参数后面**(SyntaxError)。Python 要求位置参数在前。改成全关键字 `Model(vocab_size=..., n_embd=..., block_size=...)` 或全位置(按定义顺序)。

### Q：变量名写重了(两个 self.token_embedding_table)？
第二个把第一个**覆盖**了。位置表要叫 `self.position_embedding_table`。OOP 里同名属性后者覆盖前者，不会报错但逻辑全错——容易漏。

### Q：`q = self.query` 不对？
`self.query` 是层(Wq矩阵)本身，要 `self.query(x)` 才是 `x@Wq` 的结果。`nn.Linear`/`nn.Module` 都要像函数一样**调用**(传入x)才会执行计算。
